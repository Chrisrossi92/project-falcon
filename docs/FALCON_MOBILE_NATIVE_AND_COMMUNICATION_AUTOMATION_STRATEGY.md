# Falcon Mobile, Native, And Communication Automation Strategy

## Purpose

This addendum records Falcon's long-term mobile, PWA, native application, and communication
automation strategy without expanding MVP scope.

Falcon remains web-first for MVP. During active MVP development, Falcon should stay mobile-safe and
architecturally compatible with future PWA and native application surfaces. The future native app
should be possible because Falcon's runtime authority is API/RPC-first, not because the MVP pauses
for app-store work.

This is planning documentation only. It does not change runtime behavior, routes, shells,
permissions, backend behavior, Supabase behavior, Vercel behavior, environment variables, CSP,
workflow behavior, email delivery, notification delivery, PWA behavior, native application
behavior, or production data.

## Strategy Boundary

Falcon's mobile/native posture is:

- **Web-first during MVP:** keep the browser application as the primary product surface.
- **Mobile-safe during active development:** every current workspace should remain responsive,
  touch-safe, and readable on practical mobile viewports.
- **PWA-compatible during and after MVP:** do not make architectural choices that block an
  installable web app later.
- **Native-app compatible by architecture:** keep business authority in governed APIs/RPCs so a
  future native app can use the same contracts.
- **Native app future/post-MVP:** do not treat native implementation as an MVP blocker.
- **Communication automation future platform capability:** record the doctrine now, but do not
  implement automation until workflow, notification, template, audit, and rate-limit authority are
  explicitly designed.

## Product Surface Model

### Desktop = Mission Control

Desktop should remain Falcon's best surface for dense operational control:

- owner/admin operational command;
- active order triage;
- workload and review queue visibility;
- Team Access and invitations;
- Owner Setup and readiness;
- client relationship management;
- detailed Order, Client, Assignment, Calendar, Saved Views, and Historical Orders work;
- reports, compliance, export, and future analytics.

Desktop can carry more context, compare more objects, and support deeper setup tasks. It should not
force every role to think like an owner/admin, but it is the natural mission-control surface.

### Mobile = Operational Execution

Mobile should focus on fast, role-native work execution:

- appraiser assigned work and due-date checks;
- reviewer review queue checks and decision prompts;
- assignment/vendor received work and offer response;
- owner/admin exception triage;
- quick status confirmations;
- field notes and document/photo capture later, only after storage and permission governance is
  ready;
- lightweight alert handling and acknowledgement.

Mobile should not mirror every desktop admin surface. It should reduce cognitive load by showing
the role's next action first.

## Architecture Principles

Future mobile, PWA, and native surfaces must preserve Falcon's authority model:

- permissions, route guards, RLS/RPCs, object visibility, and workflow state remain canonical;
- UI shells choose emphasis, language, and action placement, not authority;
- business logic should stay behind governed query wrappers, APIs, RPCs, and backend-owned actions;
- direct frontend table writes, local-only workflow authority, and native-only permission bypasses
  remain disallowed;
- role and shell resolution should become reusable enough for web, PWA, and native clients;
- object contracts should avoid desktop-only assumptions;
- mobile/native clients should call the same approved workflow and document paths as the web app;
- audit/activity/notification records should remain generated by authoritative backend or governed
  application paths, not duplicated by client-specific shortcuts.

The guiding test is simple: a native app should be able to exist later without reimplementing
Falcon's authority model.

## Mobile-Safe Frontend Governance

Mobile-safe UX is active governance now, even though PWA and native implementation are deferred.

Current and future web work should preserve:

- responsive workspace layouts;
- touch-safe buttons and primary actions;
- no hover-only critical controls;
- stable headers, filters, tables, and cards at mobile widths;
- clear loading, empty, and error states on narrow screens;
- role-native mobile priority, not a compressed owner/admin console;
- confirmation clarity for high-impact actions;
- readable order, review, assignment, calendar, client, and team context;
- no raw storage paths, tokens, internal permission keys, or backend implementation terms in mobile
  copy;
- no mobile-specific bypass around existing route, permission, workflow, storage, or assignment
  boundaries.

Mobile work should be judged by whether a user can answer: "What needs my attention, and what can I
safely do next?"

## PWA Direction

PWA is the likely intermediate step between MVP web and any future native app.

Possible future PWA goals:

- installable web app shell;
- responsive role-centric dashboard entry;
- safe auth/session behavior;
- mobile-friendly quick action surfaces;
- optional notification integration after notification governance exists;
- careful offline/caching review only after data sensitivity and stale-state behavior are designed.

PWA non-goals for now:

- no service worker implementation in this slice;
- no push notifications in this slice;
- no offline workflow behavior in this slice;
- no PWA-only workflow logic;
- no weakening of CSP, auth, storage, or Supabase target evidence requirements.

## Native App Direction

Native app work is future/post-MVP. It should start with operational execution, not full desktop
parity.

Likely first native-compatible surfaces:

- Appraiser `My Work`;
- Reviewer `Review Queue`;
- Assignment/Vendor `Received Work`;
- Owner/Admin exception triage;
- role-aware alerts;
- quick status confirmations;
- safe document/photo capture after storage, Edge Function, and document authority are proven;
- simple activity or note capture where governed paths already exist.

Deferred native surfaces:

- full Owner Setup;
- Team Access administration;
- deep reporting/analytics;
- broad client relationship management;
- advanced Saved Views management;
- historical/admin readback beyond simple access if needed;
- complex workflow administration;
- template and automation rule editing until the desktop administration model exists.

## Communication Automation Engine

Communication automation is a future platform capability, not an immediate MVP blocker.

The purpose is to help Falcon prompt people when operational state needs attention while reducing
noise when users keep work current. Automation should reward good operational hygiene: if an order,
review, assignment, document request, or status is already up to date, Falcon should suppress
unnecessary reminders.

Automation must remain:

- permission-governed;
- role-aware;
- state-driven;
- owner/admin configurable;
- auditable;
- logged;
- rate-limited;
- non-spammy;
- reversible or safely disableable;
- clear about why a message was sent or suppressed.

Automation must not become hidden workflow authority. It can prompt, remind, escalate, summarize, or
confirm through approved actions, but workflow state must still change only through governed paths.

### Smart Reminder Suppression

Reminder suppression should be a first-class feature, not an afterthought.

Suppress or reset reminders when safe evidence shows the work is current, such as:

- an appraiser submitted or resubmitted work;
- a reviewer cleared review or requested revisions;
- a vendor accepted, declined, submitted, or updated received work;
- an owner/admin manually excluded a reminder path;
- a relevant note or activity event was recorded recently;
- a required document was uploaded or archived through approved paths;
- a due date, assignment state, workflow state, or client-facing state changed.

Suppression rules should prevent:

- duplicate email, in-app, and future push reminders for the same event;
- reminders after a user already took the expected action;
- noisy owner/admin alerts for every visible event;
- repeated escalation without owner/admin-configured thresholds;
- reminders to users who do not have the required visibility or responsibility.

### Owner/Admin Configurable Rules

Future owner/admin automation configuration should cover:

- reminder timing;
- escalation thresholds;
- recipient roles;
- recipient exclusions;
- client/company exclusions;
- quiet hours;
- allowed channels;
- automation enable/disable by event type;
- template selection;
- owner/admin override behavior;
- audit visibility for sent, skipped, suppressed, and failed messages.

Configuration should never expose secret values or backend implementation internals.

### Editable Email Templates

Editable email templates are future work. They should support operational clarity without allowing
unsafe data exposure.

Future template requirements:

- owner/admin editing where permissioned;
- approved variable/placeholders only;
- preview and test-send behavior;
- template version history;
- audit logging for template edits;
- clear fallback templates;
- no raw SQL, tokens, storage paths, private payloads, or unrestricted object fields;
- role-aware and client-safe template variants where needed.

Templates should be operational and restrained. They should not create a marketing automation
system inside MVP Falcon.

### Role-Aware Quick Status Confirmation Actions

Future mobile, PWA, native, and email-driven quick actions may become valuable only if they call
existing governed workflow paths.

Candidate quick confirmations:

- confirm site visit scheduled or completed;
- acknowledge revision request;
- submit or resubmit to review;
- clear review or request revisions where reviewer permission allows;
- accept or decline a work request;
- submit received work;
- acknowledge owner/admin escalation;
- confirm requested document received where the document model supports it.

Quick actions must be:

- permission-checked;
- object-scoped;
- explicit about consequences;
- protected against accidental repeat submission;
- logged;
- rate-limited where message-driven;
- consistent with desktop workflow behavior.

### Operational Status Signal Evidence

`docs/OPERATIONAL_EXECUTION_IMPROVEMENT_PLAN.md` defines the first operational status-signal
doctrine in Phase 1F. Future mobile, PWA, native, and automation work should treat those signals as
evidence for presentation, suppression, and escalation planning, not as lifecycle authority.

Signal evidence should be separated into:

- **derived signals:** appointment not scheduled, appointment scheduled, waiting in review,
  revisions open, stale/no loaded update, overdue with no recent loaded update, assignment offer
  awaiting response, or submitted assignment awaiting owner review;
- **explicit user signals:** inspection complete, waiting on borrower/client, work on track,
  extension may be needed, extension requested, or actively working;
- **governed workflow signals:** submit/resubmit to review, request revisions, clear review, ready
  for client, complete, accept/decline assignment offer, submit received work.

Future reminder suppression should rely on auditable evidence such as explicit status-signal
events, governed workflow events, document uploads, assignment responses, or owner/admin exclusions.
Derived silence should increase attention cautiously, but it should not prove fault or trigger
unbounded reminder delivery.

Mobile quick confirmations should remain small and operational:

- distinguish `work is on track` from `submit to review`;
- distinguish `inspection complete` from final report completion;
- distinguish `waiting on borrower/client` from client-safe portal status;
- distinguish `review active` from `clear review`;
- distinguish `assignment update` from `submit work`.

These distinctions keep future compact/mobile actions useful without creating a parallel workflow
system.

## Role Impact

### Owner/Admin

Desktop should remain the broad operational command surface. Mobile should focus on exceptions,
approval needs, due risk, review bottlenecks, assignment response state, and high-signal alerts.

Owner/admin automation should be configurable and inspectable, but owner visibility should not mean
owner notification overload.

### Staff Appraiser

Mobile should prioritize assigned orders, due dates, revision requests, site visit context, files,
notes, and submit/resubmit actions. Appraisers should not be forced through owner/admin setup or
team-management language to do field work.

### Reviewer

Mobile should prioritize review queue decisions, overdue review work, revision history, submitted
work context, and clear review/request revision actions. Reviewer reminders should focus on review
responsibility, not broad operational status.

### Assignment / Vendor Recipient

Mobile should prioritize received work, offers, due dates, instructions, owner review state, and
submit-work actions. Recipient language should use `Work Request`, `Received Work`, `Offer`, and
`Assignment` before `packet`.

### Future Client Portal User

Future client mobile/PWA/native surfaces should be request/status/document/report native. Client
automation should be sparse, client-safe, and focused on action needed, document/report
availability, and meaningful status milestones.

## Integration With Role-Centric Shell Architecture

`docs/FALCON_ROLE_CENTRIC_OPERATIONAL_SHELL_ARCHITECTURE.md` defines one platform with multiple
role-native shells. This addendum extends that doctrine across device classes:

- desktop shell priority should favor mission control;
- mobile shell priority should favor operational execution;
- PWA should reuse web contracts and role shell doctrine;
- native should reuse governed API/RPC contracts rather than fork authority;
- role-specific labels should be chosen with mobile surfaces in mind;
- command and navigation language should translate cleanly from desktop to compact mobile contexts;
- future communication automation should respect the same role, permission, and object-visibility
  model as the shells.

## Current Non-Goals

- no native app implementation;
- no PWA implementation;
- no service worker implementation;
- no push notification implementation;
- no email automation implementation;
- no editable template implementation;
- no automation rule builder;
- no route, shell, navigation, or dashboard implementation;
- no workflow, permission, RLS, RPC, Supabase, Vercel, env var, or CSP changes;
- no production data changes.
